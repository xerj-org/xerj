02·METHODOLOGY

DESIGNED TO BE
HARD TO GAME.

The harness is demo/playbooks/bench-matrix.mjs — one file, Node builtins only, checked into the repo. Its rules:

READ TRANSPORT
Read / mixed / kNN latencies are timed through a lean keep-alive HTTP client — Node core http + a shared per-host http.Agent (maxSockets:256) keyed only by host:port — applied identically to both engines. We do not use Node's global fetch/undici: it adds ~1.5 ms of pure client overhead per request that swamps both engines' sub-millisecond server times. Ingest / mixed / kNN bulk load uses curl, identically for both.
OPEN-LOOP LOAD
Read requests fire on a fixed 200 req/s cadence at t0 + i/rate, independent of when earlier responses return. A slow engine cannot slow the clock down and flatter its own tail.
SAMPLING
Per family: 15 untimed warmup calls, then p50 over 120 timed iterations. Each request has a 15 s bound — an engine that hangs is recorded as collapsed and scored, not silently dropped.
FEASIBILITY PROBE
Every family is probed first. A 4xx classifies it unsupported for that engine; the probe's result signal (hit totals, agg shape) is captured for the correctness check.
RESULT-SIGNAL MISMATCH
If the two engines return materially different results (e.g. 0 hits vs 277,449), the row is scored N/A mismatch — an engine that returns wrong or empty results cannot win on latency.
IDENTICAL WORK
track_total_hits: true is injected into every _search body on both sides. ES caps hit totals at 10,000 by default; forcing exact totals means neither engine wins by short-circuiting the count.
VERDICTS
Ratios are normalized so >1× always means XERJ is better (lower latency, higher docs/s, smaller disk, higher recall). Any LOSE row makes the runner exit non-zero — the scorecard is a CI gate, not a brochure.
HOW WE CAUGHT OUR OWN CLIENT·THE UNDICI OVERHEAD BAND · 2026-07-06

Runs 7 and 8 (47- and 53-win) timed reads through Node's global fetch, which is backed by undici. That client adds roughly 1.5 ms of fixed overhead to every request — connection bookkeeping, header parsing, promise plumbing — and both XERJ and ES answer most reads in well under a millisecond. The client overhead was larger than either engine's server time, so it drowned the signal: ~40 rows landed in a noise band where win/lose flipped on client jitter rather than engine behaviour. We A/B'd the two transports with the identical client hitting both engines (size:0 average over 300k requests):

undici fetch → XERJ 1.61 ms · ES 2.77 ms
http keep-alive → XERJ 0.126 ms · ES 0.283 ms

The keep-alive client reveals each engine's true server round-trip — and the ~2.2× gap the client overhead had been hiding. We verified the fix does not quietly favour XERJ by re-running the whole matrix through an independent curl client: XERJ was ≤ ES on every op, and ES's reused connections were not penalized. We found our own benchmark client inflating both engines' latencies, and we corrected it. That is the point of printing this section: a benchmark that can embarrass you is the only kind worth trusting when it doesn't.

HOW WE ALMOST FOOLED OURSELVES BEFORE·THE QUERY-CACHE MIRAGE · 2026-07-01

An earlier head-to-head showed XERJ winning reads 1.3–2.2× — and the numbers were a mirage. Those benchmarks repeated the same query against a static index, so XERJ's result cache served every call after the first. We were measuring cache hits, not query execution: uncached, a match_all size:10 actually took 2.28 seconds, because hit materialization scanned every matching document instead of the top from+size. We published the finding (demo/playbooks/CRITICAL_FINDING_read_perf_cache_mirage.md), fixed the O(N) path to O(from+size), and hardened the harness with the mismatch detection and identical-work rules above. Two self-corrections, both published — because the scorecard is only worth anything if we treat our own errors the way we'd treat a competitor's.

04·REPRODUCE IT

FOUR COMMANDS.
YOUR MACHINE.

Everything on this page regenerates from the repo. No hosted harness, no private dataset, no hand-tuned engine flags. 

$ git clone https://github.com/xerj-org/xerj && cd xerj
$ cargo build --release --manifest-path engine/Cargo.toml
$ bash scratchpad/es_up.sh
$ bash scratchpad/run_scorecard.sh --docs 100k,1m --clients 1,8 --knn --mixed
es_up.sh
Downloads the official Elasticsearch 8.13.4 linux-x86_64 tarball (cached after the first run), writes a single-node config with security off on port :9201, boots it with a 4 GB heap, and polls until it answers 200. Idempotent — re-running is a no-op if ES is already up.
run_scorecard.sh
Boots the release XERJ binary on :9200 with a fresh data dir, keeps it alive for the whole run, then executes the matrix against both engines and shuts XERJ down. Exits non-zero if any row is a LOSE.
HARNESS
demo/playbooks/bench-matrix.mjs — the matrix runner and scorecard generator (Node 24, no dependencies). Reads are timed through a lean keep-alive HTTP client (Node core http + a shared per-host http.Agent, maxSockets:256), applied identically to both engines; ingest / mixed / kNN bulk load uses curl, identically for both. Output lands in demo/playbooks/SCORECARD.md, which is the exact file this page's table is rendered from.

Your absolute numbers will differ with hardware; the ratios and verdicts are the claim. If your run disagrees with this page, file an issue with your SCORECARD.md — that is precisely what the harness is for.

05·KNOWN ISSUES & WHAT'S NEXT

THE RED CELLS
ARE THE ROADMAP.

Mixed read-under-write — 5 rows

Under a concurrent 8-client bulk-write stream, XERJ's read p99 still loses: match_all / bool / range / terms / cardinality fired while the writer runs land at 76–171 ms against ES's quiesced 2–17 ms. This is a genuine XERJ weakness, not a client artifact: read tail latency degrades under sustained flush/merge write pressure. It is worth noting XERJ was also carrying the heavier write load in this run (~113k vs ~76k docs/s) — it was doing more work while being read — but that does not excuse the tail. None of these 5 rows is scored in XERJ's favor, and shrinking the read-under-write tail to ES's steady-state is the top active work item.

The marginal near-tie band

A cluster of losses is a handful of aggregations and queries where both engines answer in well under 1.5 ms and ES edges XERJ by tenths of a millisecond — simple_query_string, aggs range, filter, global, composite, terms+avg(cost), pipe serial_diff, and deep from+size. These are per-request-overhead deltas, not algorithmic gaps, and shaving that fixed floor is under active profiling.

Ingest at 8 clients — one real cell left

Ingest is now a WIN in 3 of 4 cells: single-client throughput wins at both corpus sizes (100k 1.44×, 1M 1.56×) and 8-client at 100k wins 1.34×. The one remaining ingest loss is 1M × 8 clients: 338k docs/s to ES's 366k (0.93×). Profiling points at a flush-finalize CPU ceiling — segment finalization serializes on a single core while writers wait. The fix is in flight (demo/playbooks/BEAT_ES_MASTER_PLAN.md). Definition of done is unchanged: every cell in the scorecard green, enforced by CI — any new LOSE fails the build.

Production hardening landed along the way

The perf work did not come at the expense of durability — the opposite: mappings now persist durably across restarts; a merge-race segment-lease mechanism keeps in-flight queries pinned to live segments (verified with 601k queries under continuous merge churn — zero mismatches); WAL and segment garbage is cleaned up instead of accumulating; and a critical merge data-loss defect was found and fixed along the way. This run adds a dedicated, deprioritised ingest pool so bulk writes can no longer occupy the query executor, cooperative query timeouts so a slow query yields instead of wedging its worker, and a fix for a quadratic FTS segment-write path.

Families this corpus can't measure

Skipped, not hidden — each needs a purpose-built index the flat telemetry corpus lacks: geo_* queries and aggregations (no geo_point/geo_shape field), ip_range / ip_prefix (no ip field), nested / has_child / has_parent (flat corpus, no join mapping), span_* (needs a positional text field), significant_text (corpus fields are keyword), semantic / hybrid retriever (needs a dense_vector field — kNN is covered separately by --knn on a purpose-built index), and percolate (parses but no-ops — not benchmarkable for correctness). Purpose-built corpora for these families are planned follow-ups.

The standing guardrail

Every performance change must hold ES-YAML REST conformance at 1326 passed / 0 failed. Speed bought with correctness is not a win, and it does not merge.
