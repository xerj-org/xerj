THE COST OF ELASTICSEARCH

  • Memory is the bottleneck. A production cluster spends most of its budget on RAM for the JVM heap, not for your data. Scale to more indices and you're buying heap, not capacity.
  • Specialized engineers required. ES needs dedicated operators who understand shard allocation, heap sizing, GC tuning, ILM rollover, and cross-cluster replication.
  • GC pauses cause p95 spikes. JVM stop-the-world pauses inject latency spikes that show up in your SLOs and your on-call pages.
  • Feature-gated pricing. SSO, RBAC, ML anomaly detection, and cross-cluster search are locked behind Platinum+ licensing. The enterprise bill is opaque and non-negotiable.
  • Ecosystem tax. Running ES means also running Kibana, Logstash, and Beats — services to operate before your first query.

THE XERJ ANSWER

  • Drop-in API compatibility. Port 9200, same endpoints: POST /:index/_search, POST /_bulk, POST /_aliases. Existing ES client libraries work unchanged.
  • No JVM, no GC pauses. No heap to size, no stop-the-world stalls. Predictable latency at every percentile, at a fraction of the memory footprint.
  • Smaller on disk. Domain-aware columnar compression with automatic encodings — 672.5 MB vs 806.7 MB on the same corpus (1.20× smaller).
  • One binary replaces five. No Kibana, no Logstash, no Beats, no separate vector DB. REST API + Prometheus metrics + native ingest — all in-process.
  • All features, one tier. Auth, TLS, RBAC, aggregations, vector search, hybrid queries — everything ships in the binary. No platinum paywall.
