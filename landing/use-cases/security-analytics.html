THE ELASTICSEARCH PROBLEM

  • Slow aggregations kill investigations. A terms agg on source_ip across 1M events takes 29.8 ms on ES. In a live incident, that delay compounds across dozens of pivots.
  • Cluster complexity adds ops burden. A production ES SIEM cluster needs 4+ nodes, dedicated master election, shard tuning, and a team that understands JVM heap sizing. That team costs $150K+ per engineer.
  • Cost scales linearly with retention. Every day of retention adds shards, replicas, and heap. Compliance says "keep 90 days"; finance says "cut nodes".
  • Vendor lock-in via licensing tiers. ES paywalls RBAC, SSO, and ML-based detection behind Platinum+. The features your SOC needs most are the ones that cost the most.

THE XERJ ANSWER

  • 74× faster aggregations. The same top-source-IP query runs in 0.4 ms — sub-millisecond, every time, without warm-up.
  • One binary, zero cluster ops. No master election, no shard calculator, no JVM tuning. A single xerj binary replaces the 4-node ES cluster.
  • 21× less memory at idle. 400 MB RSS vs 8.5 GB across 4 ES nodes. That translates directly to smaller instances and lower cloud bills.
  • All features in one tier. RBAC, API-key auth, TLS, aggregations, hybrid search — every feature ships in the binary. No Platinum gate.
  • 90-day retention is one config line. [logs] retention_days = 90. No ILM policies, no rollover aliases, no curator scripts.
