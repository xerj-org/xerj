THE ELK STACK TAX

  • Four services to run before you see a log. Filebeat → Logstash → Elasticsearch → Kibana. Each has its own config, its own memory footprint (Logstash alone: 500 MB–2 GB), and its own failure modes.
  • ILM policies are fragile. Index lifecycle management in ES involves rollover aliases, warm/cold/frozen tiers, curator scripts, and a naming convention that breaks when someone creates an index manually.
  • Cold start kills your morning. After a restart, ES takes ~15 seconds of JVM warm-up before the first query returns. In an incident, 15 seconds is a lifetime.
  • Shard tuning is a full-time job. Too many shards per index → master bottleneck. Too few → uneven distribution. The "right" number changes with data volume, and there's no undo button.

THE XERJ ANSWER

  • One binary, four ingest paths. turbo-ingest, structured logs, syslog, and OTLP — all in-process. No Logstash, no Beats.
  • Retention is one number. [logs] retention_days = 30. Time-partitioned internally so pruning is O(partitions), not O(documents).
  • 300× faster cold start. 50 ms to first query vs ~15 seconds ES JVM warm-up. WAL replay is fast because it's append-only and the format is simple.
  • No shard tuning. XERJ manages segments internally — you never configure shard count, replica count, or allocation rules for log data.
