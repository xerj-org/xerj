# XERJ vs Elasticsearch — Scorecard

Generated by `demo/playbooks/bench-matrix.mjs`. Identical workload, same machine (localhost), single node, security off. Corpus: real LLM-telemetry (100k docs for reads). docs = {100k}, clients = {1}.

Verdict is from XERJ's POV: WIN = XERJ better (lower latency / higher docs·s / smaller disk / higher recall). Ratio is normalized so **>1× means XERJ is better**. An *unexpected* LOSE fails CI; the four `mixed … (under write)` p99 cells below are a **known, documented architectural gap** (live-memtable reads under the writer's per-shard lock — see `MIXED_READ_UNDER_WRITE_FINDING_2026-07-08.md`) and are tracked separately, not CI-gating.

> **Methodology — read latency transport.** Read/mixed/kNN latencies are measured with a lean **keep-alive HTTP client** (Node core `http` + a shared per-host `http.Agent`, `maxSockets: 256`), applied **identically to both engines** for every operation and keyed only by `host:port` (both localhost). We do **not** use Node's global `fetch`/undici: it adds ~1.5ms of pure client overhead per request, which swamps the sub-millisecond server times of *both* engines and compresses every read into a noise band. Measured with the identical client hitting both engines (`size:0` avg over 300k): undici fetch → XERJ 1.61ms, ES 2.77ms; http keep-alive → XERJ 0.126ms, ES 0.283ms. The keep-alive client reveals each engine's **true server round-trip** (the ~2.2× gap the client overhead had hidden), not the Node client. Ingest/mixed/kNN bulk load uses `curl` identically for both engines.

> **Methodology — uncached execution (honesty).** Every read is measured with `request_cache=false` and XERJ run with `XERJ_DISABLE_QUERY_CACHE=1`, so BOTH engines EXECUTE every query on every iteration (no whole-result cache clone). Query bodies are never mutated per-iteration; lower-level filter/OS caches stay on for both. Read rows show p50 (verdict) with p99 alongside. Latency rows within a noise band (`|Δ| ≤ max(0.30ms, 20%)`) score **TIE** (does not fail CI); only **LOSE** fails CI.

> **Methodology — mixed read-under-write (iso-load).** The background writer is OPEN-LOOP: one `BATCH`=10k `_bulk` launched every 250ms on an absolute-deadline schedule (offered **40,000 docs/s**), byte-identical for both engines, independent of how long each bulk takes (a closed-loop writer would offer LESS load to a stalling engine exactly when it stalls). At most 3 bulks in flight; a tick finding >2 in flight skips and counts an **overrun**. Every mixed row is annotated with the writer mode + achieved rates; iso-load is declared only when BOTH engines achieve within ~5% of the offered target. Rows from older CLOSED-LOOP runs are not comparable to open-loop rows.
> XERJ: mode open-loop, offered 40,000/s, achieved 39,626/s (2,080,000 docs launched, 0 overrun ticks), end-of-window visible docs 2,380,000 (each engine's own visibility semantics: ES `_count` excludes unrefreshed docs — it is not memtable depth).
> ES: mode open-loop, offered 40,000/s, achieved 39,688/s (2,090,000 docs launched, 0 overrun ticks), end-of-window visible docs 2,370,000 (each engine's own visibility semantics: ES `_count` excludes unrefreshed docs — it is not memtable depth).

## Skipped families (need a purpose-built index the flat corpus lacks)

- `geo_distance / geo_bounding_box / geo_polygon / geo_shape` — no geo_point/geo_shape field in corpus
- `geohash_grid / geotile_grid / geo_distance-agg / geo_bounds / geo_centroid` — no geo_point field in corpus
- `ip_range / ip_prefix` — no ip field in corpus
- `nested / has_child / has_parent (join)` — corpus is flat, no nested/join mapping
- `span_term / span_near / span_or / span_not / span_first / span_containing / span_within` — needs a text field with positions
- `significant_text` — needs an analyzed text field (corpus fields are keyword)
- `semantic / hybrid retriever` — needs a dense_vector field + embed step (see --knn)
- `knn (as query/top-level)` — covered separately by --knn on a purpose-built dense_vector index
- `percolate` — now a real reverse-search doc-scan (each doc stores a query; supplied documents are matched against it — `index.rs` `QueryNode::Percolate`), but skipped here because the flat telemetry corpus has no percolator field of stored queries to match against
