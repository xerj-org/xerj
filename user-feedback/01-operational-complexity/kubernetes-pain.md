# Kubernetes and Cloud-Native Pain Points

## Severity: HIGH | Frequency: HIGH (growing with K8s adoption)

---

## Fundamental Conflicts

### Stateful on Ephemeral Platform
- "Running stateful applications such as Elasticsearch in Kubernetes is challenging because pods move frequently across nodes to balance resources, but there is no automatic way to move data along with them"
- Kubernetes treats pods as ephemeral; ES nodes hold state that can't be rescheduled

### vm.max_map_count Requires Privileged Containers
- ES requires `vm.max_map_count=262144` -- must be changed on the HOST, not in container
- Standard workaround: privileged init container running `sysctl -w`
- "This requires the ability to run privileged containers, which is likely not the case on many secure clusters"
- Conflicts with PodSecurityPolicies: "spec.initContainers[0].securityContext.privileged: Forbidden"

### JVM Memory Exceeds Container Limits
- Even with heap at 512MB on 1.5GiB container, "memory usage sometimes exceeds 1.5GB and crash"
- ES uses memory beyond JVM heap (off-heap, mmap, direct buffers)
- If you don't set resource.limits, ES "assumes it has access to all memory available on the K8s node"

## Operator Bugs (ECK)

### Stale State Causes Accidental Scaledowns
- After operator restart, it connects to different apiserver with stale view
- Sees old `count=1`, triggers unintended node shutdown and data migration
- GitHub #5274

### ECK Operator OOMKilled on Large Clusters
- Controller-runtime framework gathers info about ALL resources in cluster
- Default pod limit only 1Gi; fails on clusters with hundreds of resources

### Operator Upgrades Trigger Cascading Restarts
- "If you have a large ES cluster, the rolling restart triggered during operator upgrades could cause performance degradation"
- Must plan for upgrade time of operator + rolling ALL managed clusters

### Copying Secrets Can Delete Data Volumes
- "Copying ES Secrets generated by ECK into another namespace wholesale can trigger a Kubernetes bug which can delete ALL Elasticsearch-related resources, including data volumes"

## Autoscaling Failures

### Zone-Awareness Blocks Scaling
- Zalando: autoscaling "got stuck running 7 nodes" (should have been 6)
- ES refused to relocate shards because pod "was the only instance in eu-central-1a"
- Operator drain retry: "keeps retrying 999 times before giving up"
- If interrupted mid-process, "cleanup phase is never executed and node stays in exclusion list forever"

### Autoscaling Requires Enterprise License
- ECK autoscaling requires paid Enterprise license
- Only scales based on disk size, not CPU load
- Volume size cannot be scaled down
- Only one PVC per node supported

### VPA Is Impractical
- "Difficult to apply VPA to Elasticsearch since it's a stateful application requiring pod restarts"
- Standard HPA also problematic for stateful workloads

## Storage Issues

### VolumeClaimTemplates Cannot Be Modified
- "Not possible to update VolumeClaimTemplates of existing StatefulSet"
- Resizing disk = create new StatefulSet + migrate data (highly disruptive)

### Local PVs Pin Pods to Nodes
- Once bound to local PV, Pod can only run on same host
- If host goes down, Pod stays Pending until host returns or PVC manually deleted

### EBS Burst Credit Exhaustion
- What appears as OOM is actually storage: "EBS burst credit decreased slowly before crash. Once it reaches 0, pod gets evicted"
- GC warnings mask the real cause (disk I/O starvation)

## Networking and Performance

### Added Latency vs Bare Metal
- "Response time being slower in EKS than in bare Docker, even with better hardware (12000 IOPS vs 3000 IOPS)"
- ES is "chatty service with frequent inter-node communication"

### JVM DNS Caching Breaks Master Discovery
- JVM caches DNS resolutions indefinitely with security manager
- When master pods recreated with new IPs, data nodes connect to old cached IPs
- Fix requires custom security properties file (can't set via JVM options)

### Rolling Restarts Cause Write Unavailability
- "Loss of write availability for a period of 10 mins" during normal rolling restart
- Shard recovery during restarts adds unnecessary I/O

## Deployment Pain

### Helm Charts Archived, Production-Unsafe Defaults
- Official Helm chart repository "archived on May 16, 2023 and is now read-only"
- Had "seven missing config settings that would have caused problems for production"
- Missing livenessProbe, missing resource requests, split-brain risk

### 6+ Hour Deployment Times
- One team: "deploying a single cluster taking roughly 90 minutes, with four clusters total deployment time stretched beyond 6 hours"

### GKE Private Clusters Need Manual Firewall Rules
- ECK submissions timeout because webhook can't be reached
- Must manually add firewall rule for port 9443
