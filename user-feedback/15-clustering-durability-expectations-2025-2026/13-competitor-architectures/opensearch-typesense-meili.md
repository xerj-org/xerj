## Summary of Key Architecture Findings

### OpenSearch
- Lucene-based, Java/JVM, full Elasticsearch-compatible cluster model
- Split brain risk requires minimum 3 master-eligible nodes
- Shard count is immutable post-creation — major planning constraint
- JVM heap pressure frequently causes node crashes in production
- 2-node clusters guaranteed to enter unrecoverable quorum loss state (requires AWS Support, 24–72h outage)
- Rolling upgrades can cause cluster RED state during shard allocation window
- Unassigned shards / stuck initialization are endemic operational problems
- Operational burden estimated at 50% of one FTE for a 25-node cluster
- AWS added Cluster Insights (Nov 2025) to address pre-existing observability gap

### Typesense
- C++ in-memory engine, Raft consensus built-in (open source)
- All 3 nodes in HA hold full dataset in RAM — memory doesn't scale-out, it scales-up
- Restart requires full in-memory index reconstruction: hours of downtime for large datasets
- No hard memory cap — OOM causes process crash; swap recommended as mitigation
- Write amplification (50–100x) on large datasets before config tuning
- 24-hour lag on cloud capacity auto-scaling
- Self-hosted HA networking complexity (ports 8107/8108, security groups, TLS)
- Schema changes require full re-index

### Meilisearch
- Rust, LMDB disk-mapped storage, open-source single-node
- Clustering/sharding is Enterprise Edition only (BUSL 1.1) — requires commercial license for production
- No aliasing for zero-downtime reindexing (gap vs. Elasticsearch)
- ~200 index limit in open-source (OS-dependent)
- No function scoring / complex relevance ranking in open source
- Dramatically lower memory footprint vs. Elasticsearch (290MB vs 1.3GB for same dataset)
- Easiest operational experience when single-node failure tolerance is acceptable
